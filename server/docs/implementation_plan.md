# План реализации бэкенда

Этот документ описывает пошаговый план создания бэкенда для онлайн-аптеки на основе предоставленной документации.

## Фаза 1: Настройка проекта и основы архитектуры

Цель: Подготовить структуру проекта, настроить окружение и заложить основу для "Чистой архитектуры".

1.  **Настройка структуры проекта:** [ВЫПОЛНЕНО]
    *   Создать директории для модулей в `app/modules/` в соответствии с `architecture.md`: `auth`, `users`, `catalog`, `inventory`, `pharmacies`, `orders`, `search`.
    *   В каждом модуле создать поддиректории: `domain`, `application`, `infrastructure`, `presentation`.

2.  **Конфигурация окружения:** [ВЫПОЛНЕНО]
    *   Настроить подключение к PostgreSQL в `.env` файле.
    *   Настроить локальное хранение загружаемых изображений через Laravel `public` disk.

3.  **Создание базовых классов "Чистой архитектуры":** [ВЫПОЛНЕНО]
    *   Создать базовые классы/интерфейсы в `app/shared/` для:
        *   `Entity`
        *   `ValueObject`
        *   `DomainEvent`
        *   `Repository` (контракт)
        *   `UseCase` (контракт)
        *   `DTO` (Data Transfer Object)

## Фаза 2: Реализация основных модулей

Цель: Реализовать ключевые модули, отвечающие за пользователей, аутентификацию и каталог товаров.

1.  **Модуль `users` и `auth`:** [ВЫПОЛНЕНО]
    *   Создать миграции для таблиц `users`, `roles`, `user_roles` в соответствии с `data.md`.
    *   Реализовать сущности (Entities) `User` и `Role` в `app/modules/users/domain/`.
    *   Создать репозитории (Contracts и Eloquent-реализации) для `User` и `Role`.
    *   Реализовать Use Cases для:
        *   Регистрации (`POST /auth/register`)
        *   Аутентификации (`POST /auth/login`)
        *   Выхода из системы (`POST /auth/logout`)
        *   Получения текущего пользователя (`GET /auth/me`)
        *   Обновления профиля (`PATCH /users/profile`)
    *   Настроить Laravel Sanctum для аутентификации по токену.
    *   Реализовать контроллеры и ресурсы для всех эндпоинтов.
    *   Настроить RBAC с помощью Middleware или Policies на основе матрицы из `api.md`.

2.  **Модуль `catalog`:** [ВЫПОЛНЕНО]
    *   Создать миграции для всех таблиц каталога: `sections`, `categories`, `brands`, `manufacturers`, `products`, `product_images`, `attributes`, `product_attributes`.
    *   Реализовать сущности для всех перечисленных таблиц.
    *   Создать репозитории для каждой сущности.
    *   Реализовать Use Cases для:
        *   CRUD-операций для всех сущностей каталога (для роли `ADMIN`).
        *   Получения списков и деталей (для всех ролей).
        *   Получения популярных товаров.
    *   Реализовать контроллеры, реквесты (для валидации) и ресурсы.

3.  **Модуль `pharmacies`:** [ВЫПОЛНЕНО]
    *   Создать миграцию для таблицы `pharmacies`.
    *   Реализовать сущность `Pharmacy`.
    *   Создать репозиторий.
    *   Реализовать Use Cases и контроллеры для CRUD-операций (для `ADMIN`) и получения списка/деталей (для всех).

4.  **Модуль `inventory`:** [ВЫПОЛНЕНО]
    *   Создать миграцию для таблицы `inventory`.
    *   Реализовать сущность `Inventory`.
    *   Создать репозиторий.
    *   Реализовать Use Cases для управления остатками (для ролей `ADMIN` и `MANAGER`).
    *   Реализовать логику резервирования и списания остатков.

## Фаза 3: Реализация функциональных модулей

Цель: Реализовать модули, отвечающие за поиск, корзину, избранное и заказы.

1.  **Модуль `search`:** [ВЫПОЛНЕНО]
    *   Реализовать Use Case и контроллер для поиска товаров (`GET /search/products`) через PostgreSQL-запросы по каталогу.
    *   Поддержать простой поиск по названию, slug и описанию, а также фильтры из `api.md`.

2.  **Модуль `favorites`:** [ВЫПОЛНЕНО]
    *   Создать миграцию для таблицы `user_favorites`.
    *   Создать репозиторий.
    *   Реализовать Use Cases и контроллеры для получения, добавления и удаления товаров из избранного.

3.  **Модуль `cart`:** [ВЫПОЛНЕНО]
    *   Создать миграцию для таблицы `cart_items`.
    *   Создать репозиторий.
    *   Реализовать Use Cases и контроллеры для всех операций с корзиной (добавление, обновление количества, удаление, очистка).

4.  **Модуль `orders`:** [ВЫПОЛНЕНО]
    *   Создать миграции для таблиц `orders`, `order_items` и справочников `order_statuses`, `delivery_methods`, `payment_methods`.
    *   Реализовать сущности `Order` и `OrderItem`.
    *   Создать репозитории.
    *   Реализовать Use Case "Создание заказа" (`POST /orders`):
        *   Валидация входных данных.
        *   Проверка наличия товаров на складе.
        *   Проверка запрета на доставку рецептурных товаров.
        *   Резервирование остатков в `inventory`.
        *   Сохранение заказа и его состава в транзакции.
    *   Реализовать Use Cases и контроллеры для получения списка и деталей заказов (для пользователя и администратора), а также для отмены заказа.
    *   Реализовать Use Case для изменения статуса заказа (для `ADMIN` и `MANAGER`).

## Фаза 4: Завершение и тестирование

Цель: Обеспечить качество кода, покрыть его тестами и реализовать оставшуюся функциональность.

1.  **Загрузка файлов:** [ВЫПОЛНЕНО]
    *   Реализовать Use Case и контроллер для загрузки изображений товаров (`POST /admin/products/{id}/images`) с использованием локального `public` disk.
    *   Реализовать удаление изображений.

2.  **Тестирование:** [В ПРОЦЕССЕ]
    *   Написать Unit-тесты для Domain и Application слоев (бизнес-логика, Use Cases).
    *   Написать Feature-тесты для всех эндпоинтов API, проверяя корректность ответов, статусов и прав доступа.

3.  **Документация и ревью:** [В ПРОЦЕССЕ]
    *   Убедиться, что реализация соответствует `api.md` и `constraints.md`.
    *   Провести ревью кода на предмет соответствия "Чистой архитектуре" и отсутствия нарушений границ между слоями и модулями.
    *   Сгенерировать документацию API, если требуется (например, с помощью Scribe или OpenAPI).
